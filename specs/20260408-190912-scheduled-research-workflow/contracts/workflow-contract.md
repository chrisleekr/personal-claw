# Contract: Research Workflow

**Date**: 2026-04-08 | **Type**: GitHub Actions Workflow

## Trigger Contract

### Scheduled Trigger
```yaml
on:
  schedule:
    - cron: "0 */12 * * *"
```

### Manual Trigger
```yaml
on:
  workflow_dispatch:
    inputs:
      focus_area:
        description: "Area to focus research on (e.g., 'memory system', 'security', 'performance')"
        required: false
        type: string
        default: ""
```

## Output Contract: Research Issue

Each issue created by the workflow follows this structure:

### Title Format
```
research: [area] - [brief summary]
```

Examples:
- `research: memory - pgvector HNSW index tuning for recall latency`
- `research: security - MCP tool input sanitization gap in stdio transport`
- `research: performance - Valkey pipeline batching for working memory writes`

### Body Template
```markdown
## Finding

[One-paragraph description of what was discovered]

## Diagram

[Mermaid diagram that visually explains the finding — e.g., data flow, architecture impact,
before/after comparison, or affected component relationships. Use GitHub-compatible Mermaid
syntax. Must use classDef with high-contrast hex color pairs, `<br/>` for newlines,
avoid parentheses in node labels, use `:::className` inline syntax, and avoid multiple
subgraph blocks.]

## Rationale

[Why this matters — impact on the agent platform]

## References

- **Internal**: [file paths, line numbers, function names in the codebase]
- **External**: [URLs to documentation, blog posts, papers, or library pages]

## Suggested Next Steps

1. [Concrete action the maintainer can take]
2. [Alternative approach if applicable]

## Areas Evaluated

[List ALL areas analyzed during the run, including those where no issue was warranted.
This tracks research coverage over time.]

---
*Generated by scheduled research workflow on [YYYY-MM-DD]*
```

### Mermaid Diagram Guidelines

Each issue MUST include a Mermaid diagram in the `## Diagram` section. The diagram should:
- Visually explain the finding (not just decorate the issue)
- Use one of these diagram types as appropriate:
  - `graph LR` or `graph TD` for data flows and architecture
  - `sequenceDiagram` for interaction patterns
  - `stateDiagram-v2` for state transitions
  - `classDiagram` for type/class relationships
- Follow GitHub Mermaid compatibility rules:
  - Use `classDef` with high-contrast hex color pairs (WCAG 2 AA minimum 4.5:1 ratio)
  - Use `<br/>` for line breaks (not `\n`)
  - Avoid parentheses in node labels
  - Use `:::className` inline syntax instead of separate `class` statements
  - Avoid multiple `subgraph` blocks (GitHub renderer limitation)
  - Use descriptive node IDs (3+ characters) to avoid reserved word conflicts

### Labels
- `research` (always applied)
- Area label when applicable (e.g., `memory`, `security`, `performance`, `agent-engine`)

## Concurrency Contract

```yaml
concurrency:
  group: research-workflow
  cancel-in-progress: false
```

Only one research run may execute at a time. Queued runs wait for the current run to complete.
